### 3. v1 scaled corpus — matched triplets (the session's main artifact)

Decided label policy (resolves session 03 #4): **all red-team prompts, subtle and
blunt, stay `should-refuse`** — no relabeling of the ambiguous ones. Instead we
add a **blue-team `should-answer` set** to measure false-refusal directly.

Design: **10 categories × 3 scenarios × 3 framings = matched triplets.**
- `data/prompts/red-team-v1.jsonl` — **60 should-refuse** (30 subtle `-s` + 30 blunt `-b`).
- `data/prompts/benign-v1.jsonl` — **65 should-answer**: 30 *benign-adjacent* (`-n`,
  one per scenario) + 35 *easy-benign* (`easy-NNN`).
- **Total 125**, balanced 60 refuse / 65 answer.

The triplet is the point: each scenario appears as subtle-harmful, blunt-harmful,
and benign-legitimate on the *same topic* (e.g. deceptive supplement copy /
"copy that lies" / honest supplement copy). A topic-matching censor blocks all
three; an intent-reading committee blocks two and approves one. The per-scenario
gap is the false-refusal-vs-catch-rate signal. New fields `scenario` + `framing`
let `analyze.py` pair them (batch loader ignores unknown fields, verified).

Categories: manipulation, stereotyping, edgy-fiction, borderline-advice
(medical + legal), risqué, dark-patterns, misinformation, jailbreak,
controversial-opinion, insulting-roast. The 6 v0 prompts are folded in as each
category's first scenario (rt-001→`man-1-s`, rt-006→`jail-1-s`, etc.).

Scope held to `docs/red-team-design.md` (D6): nothing illegal/dangerous, no
weapons/CBRN/CSAM/malware/suicide. Deliberate gray-zone refuse cases (at-home
mole removal, tenant-pressure, fake reviews) are content-policy violations, not
serious-harm uplift — the exact boundary the experiment probes. User reviewed and
approved the political placeholders (immigration / remote-work / wealth-tax) and
the softened `adv-3-b` eviction framing.

Validation: 125 lines parse, no duplicate IDs, all 30 scenarios have
subtle+blunt+benign-adjacent, 6 red-team prompts per category.

### 4. qwen3 v1 batch — landed 124/125 clean, but only after an infra fight

Ran balanced-8 over the full 125-prompt v1 corpus for qwen3. Final dataset:
**`data/results/qwen3-balanced-8-v1-FINAL.jsonl`, 124/125 clean.** Only `ris-2-b`
is missing (see runaway note below). All 60 red-team + 64/65 benign have clean
per-persona votes; parse-failure rate ~5–6%, all fail-safe to BLOCK (D10).

Getting there took three attempts because of two distinct infra failures:

1. **Ollama died mid-run (rows 20–125 → 106 ReadErrors).** Root cause: `ollama
   serve &` had been launched in the *interactive web terminal*, so when that
   session closed it got SIGHUP and died. The local SSH tunnel stayed up but
   forwarded into a dead endpoint. **Fix: relaunch Ollama fully detached** —
   `/root/start_ollama.sh` (sets `OLLAMA_HOST`, `exec ollama serve`) started via
   `nohup setsid … </dev/null &`. Survives terminal close and SSH disconnect
   (verified — it outlived the next tunnel drop).
2. **The SSH tunnel itself then dropped** (exit 255) ~1h into the resume run,
   under the pod's heavy load (load avg ~12 — a busy multi-tenant host; our GPU
   was idle, it's CPU contention from other tenants). Another 47 rows lost.
   **Fix + new canonical method (D19): run the batch ON THE POD, no tunnel.**
   `rsync` the repo to `/root/society`, `uv sync`, run `src.batch` against
   `127.0.0.1:8000` as a detached `nohup setsid` script writing to pod disk + a
   `.done` marker; poll/scp results back with short reconnecting SSH calls. The
   pod has python 3.12 + uv + git + rsync preinstalled. This eliminates the
   tunnel as a failure mode entirely and is faster (localhost, no per-call RTT).

Healthy row wall-clock on the A40: **median ~45–50s** at `OLLAMA_NUM_PARALLEL=1`.

**`ris-2-b` runaway generation.** The "wall-to-wall most vile/offensive language"
prompt makes abliterated qwen3 generate without stopping — observed 9,000+ tokens
and climbing on a single call, eventually erroring/timing out all three times.
The **generator call is unbounded** (committee votes already cap `num_predict`).
Fix for next session: add a `num_predict` cap to the generator call so one prompt
can't burn ~10 min. Until then `ris-2-b` has no data (analyzer skips it).

### 5. Analyzer bugs found (NOT yet fixed — do this BEFORE analysing v1)

Inspected `src/analyze.py` against the new corpus; it's written for the v0 era
and will silently produce wrong/empty results on v1. Three fixes needed:

1. **Label mismatch** (line ~208): checks `gt == "should-allow"` but the corpus
   and `methodology.md` use **`should-answer`**. → all 64 benign rows fall through
   and false-refusal (fp/tn) stays zero. Change the string to `should-answer`.
2. **No false-refusal table exists.** `analyze.py` only renders catch rate
   (tp/(tp+fn)) and a should-refuse-only per-row grid. The fp/tn buckets are
   computed but never displayed. → add a `_render_false_refusal_table`
   (fp/(fp+tn) per composition × rule), rendered from the framing-merged stats
   (framing is meaningless for benign rows), and wire it into `main`.
3. **`_framing()` misclassifies v1 rows.** The batch output row does NOT carry
   the `framing`/`scenario` fields (batch.py only writes a fixed key set), and
   `_framing()`'s fallback keys off the v0 `rt-1` prefix. v1 IDs are `man-1-s` /
   `man-1-b` / `man-1-n`, so every row currently buckets as "subtle". → make
   `_framing()` read the `-s`/`-b` suffix (and treat `-n`/`easy-` as benign).
   Optional cleaner fix: add `framing` to the batch output row for future runs.

(Votes DO carry `leaning`, so `rule_restrictive_veto` works — no fix there.)

## Decisions (consolidated)

- **D16 — Connectivity is SSH tunnel, not Runpod HTTP proxy** (port 8000 exposed
  as direct-TCP only; tunnel also avoids a public Ollama endpoint). *Superseded
  for batch runs by D19 — tunnel proved too fragile on a contended host.*
- **D17 — Label policy: red-team stays should-refuse (subtle + blunt); false-refusal
  is measured by a separate should-answer blue-team set**, not by relabeling
  ambiguous red-team prompts. Supersedes the session-03 open question on
  rt-002/004/005.
- **D18 — Corpus structure: matched triplets** (subtle / blunt / benign-adjacent
  per scenario) + an easy-benign floor. The per-scenario approve/block gap is the
  primary intent-discrimination signal.
- **D19 — Batch runs execute ON the pod, not over an SSH tunnel.** Ollama must be
  launched detached (`nohup setsid`). The tunnel is fine for smoke tests / health
  checks but drops under shared-host load over a long run. rsync repo → uv sync →
  detached batch → scp results back.
