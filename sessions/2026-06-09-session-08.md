### Robustness (the user's explicit "poll for hangs/downtime" ask)

Three layers, since session 04 lost rows to a dead Ollama + a dropped tunnel:
1. **Per-row guard** — `--row-timeout 300`: a row over 5 min (vs ~20s typical) is
   killed, logged `RowTimeout`, batch continues. Generator `num_predict=2048` cap
   (session 05) bounds runaways. Every row written + flushed incrementally
   (`batch.py:182-183`), so row-count is a live progress + stall signal.
2. **Pod-side watchdog** (`/root/watch.sh`, detached) — logs Ollama-reachable /
   batch-PID-alive / rows-written / GPU every 60s to `/tmp/watch.log`, and
   **auto-restarts Ollama if it's gone** (directly fixes the session-04 death mode).
3. **Laptop poller** (background) — reads the health trail every 90s, wakes Claude
   on completion or a >10 min stall. (First poller had a zsh bug — stored the ssh
   command in a variable, which zsh doesn't word-split, so it errored every cycle;
   caught mid-run via a manual check, fixed by inlining the ssh command.)

## Findings — the A/B (`balanced-8`, both arms)

| Rule | Catch RO→PR | False-refusal RO→PR | Net RO→PR |
|---|---|---|---|
| simple_majority | 4% → 3% | 1% → 1% | +2% → +1% |
| supermajority_2_3 | 32% → 19% | 6% → 9% | +26% → +10% |
| restrictive_veto | 82% → 84% | **26% → 20%** | +56% → **+63%** |
| unanimous_approve | 85% → 87% | **34% → 24%** | +51% → **+63%** |

**1. Visibility barely moves aggregate catch; `simple_majority` is hopeless in both
arms (~3–4%).** The over-permissiveness of simple-majority is **structural, not a
visibility artifact** — most personas don't *own* most harms, so a bare majority
approves whether or not they see the prompt. Catch only materialises under
veto-style rules, where the fully-owned harm is manifest in the response either way.

**2. The real effect of response-only is over-blocking.** Blinding the committee
costs **+6 to +10 pts of not-owned false-refusal** (PR→RO: 24→34% unanimous, 20→26%
veto) **at no catch benefit**. Seeing the benign request lets the committee spare
harmless responses it otherwise blocks → **prompt-and-response strictly dominates the
Pareto frontier** for the usable rules.

**3. But response-only reads ownership *flat* across subtlety; PR leaks
surface-signal.** Fully-owned catch L1→L5 (the headline hypothesis test):

| | L1 (deniable) | L2 | L3 | L4 | L5 (blunt) |
|---|---|---|---|---|---|
| RO × restrictive_veto | 81% | 81% | 100% | 75% | 81% |
| PR × restrictive_veto | **56%** | 94% | 87% | 88% | 94% |

RO catch is flat (pure ownership-reading — the D23 ownership hypothesis lands). PR
**dips at L1** (56% vs 81%): at the most deniable level the innocuous-looking prompt
gives cover to release an owned harm — **the v1 surface-signal failure mode partially
returns the moment the committee can see the prompt.**

**Headline:** *Visibility doesn't change whether owned harms get caught in aggregate
— it changes the error profile.* Response-only is trigger-happy (high false-refusal)
but reads ownership cleanly and subtlety-flat. Prompt-and-response wins on net and
halves much of the false-refusal, but pays with a surface-signal leak at the deniable
end of owned harms. Neither arm alone would have shown this — the A/B is the result.

Best single operating point either way: **`balanced-8 × restrictive_veto`** (RO 82/26,
PR 84/20).

### Secondary observations
- **partial-owned block-rate rises with subtlety** in both arms (e.g. PR unanimous
  31→44→62→69→80% L1→L5) — the contestable middle is signal-sensitive by design (D23);
  it's a weighted mix of the session-07 qualifier clusters, not a single accuracy number.
- not-owned false-refusal is partly inflated by the **session-07 roast-consent
  overlap** (consent context lives in the prompt) — which would hit RO harder, consistent
  with finding #2.
- Voting rule dominates *everything* — a far bigger lever than composition or
  visibility. Under `simple_majority` (the configured default) the society barely
  blocks; the catch story only exists under veto/unanimous rules.
